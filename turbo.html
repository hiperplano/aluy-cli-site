turbo · opt-in

Turbo mode

Three optional boosters that run on your machine — long-term memory, context headroom and local models. You turn them on with aluy bootstrap; the agent works fine without them. What they are, how to install, what can go wrong and how to fix it.

01 · what

What turbo turns on

All three bind to 127.0.0.1 only and are opt-in — nothing comes up unless you accept it. They rise on boot if already installed; installing is the explicit aluy bootstrap step. The opposite profile, leve, installs none.

  • mem0 · :11435 — persistent memory: the agent stores facts and preferences and recalls them next session. You see, pin or forget them with /memory.
  • headroom · :8787 — context compression: condenses what's already passed and brings back only what matters, keeping the model window under control.
  • ollama · :11434 — local model runtime: a small judge (qwen2.5:0.5b) and an embedder (nomic-embed-text) for internal tasks that never leave the machine.
02 · install

Two ways to provision

After npm i -g @hiperplano/aluy-cli and aluy onboard, run one of:

$ aluy bootstrap # via the agent (default)
$ aluy bootstrap --no-agent # direct — no model needed
  • Via the agent (default) — aluy detects the OS, installs the missing prerequisites (Python, pip, venv, with sudo when needed) and the boosters, handling problems as it goes. Runs in --yolo — opting into turbo is the consent.
  • Direct (--no-agent) — deterministic path: pinned Ollama artifact + venv creation, without using a model. Requires Python already present.

Fresh machine with no model yet? Use --no-agent. The agent installer needs a working model to think — and if that model is the very Ollama you're about to install, it's circular. From rc.35, aluy bootstrap detects the model is unreachable and falls back to the direct path on its own.

Don't run aluy with sudo. The sidecars refuse to run as root, and sudo sends aluy looking for its config in /root/.aluy. Install as your user; the installer asks for sudo only on the system steps that need it.

03 · self-repair

When something's off, the agent fixes it

In the TUI, /doctor shows the live health check. If a booster is down it asks — and /doctor fix hands the repair to the agent itself: it reads ~/.aluy/logs/*.log, runs the bootstrap, installs what's missing and retries until the three are green.

/doctor # live health check — asks if a booster is down
/doctor fix # the agent reads the logs, bootstraps and repairs
04 · troubleshooting

Common problems & fixes

The per-sidecar logs (truncated each boot) live in ~/.aluy/logs/ollama.log, mem0.log, headroom.log — the first place to look.

  • mem0 ✗ · log can't open file '…aluy-mem0-server.py' — the server script isn't in the venv. Run aluy bootstrap (copies it). On rc ≥ 33 the boot also copies it on launch.
  • mem0 ✗ · log KeyError: '_type' / no such column — store from an older version. mv ~/.aluy/memory{,.bak}; mv ~/.mem0/history.db{,.bak} and reopen. On rc ≥ 29 the server self-heals.
  • ollama ✓ but ollama list → command not found — service runs on :11434 but the binary isn't on PATH. rc ≥ 30 symlinks it into ~/.local/bin; or open a new shell.
  • bootstrap stuck at verificando ollama + erro de broker — agent install with no reachable model. Use aluy bootstrap --no-agent (rc ≥ 35 falls back automatically).
  • headroom ✗ · log No module named 'fastapi' — venv installed headroom-ai without the [proxy] extra. ~/.aluy/hr-venv/bin/pip install 'headroom-ai[proxy]==0.25.0'. Fixed at the source in rc ≥ 36.
  • all three ✗ on a new machine — nothing provisioned yet. aluy bootstrap --no-agent.
05 · manual

Provision by hand

If you'd rather set it up yourself (or understand what bootstrap does). Everything lives under ~/.aluy/.

# ollama
$ curl -fsSL https://ollama.com/install.sh | sh
$ ollama pull qwen2.5:0.5b
$ ollama pull nomic-embed-text
# mem0 — Python ≥ 3.10; needs ollama
$ python3 -m venv ~/.aluy/mem-venv
$ ~/.aluy/mem-venv/bin/pip install mem0ai==0.1.76 chromadb==0.5.23 ollama==0.4.7
$ cp "$(npm root -g)/@hiperplano/aluy-cli/assets/mem0/aluy-mem0-server.py" ~/.aluy/mem-venv/
# headroom — Python ≥ 3.10; [proxy] brings fastapi
$ python3 -m venv ~/.aluy/hr-venv
$ ~/.aluy/hr-venv/bin/pip install 'headroom-ai[proxy]==0.25.0'

After provisioning by hand, aluy doctor should show the three green (boot brings them up when the profile is turbo). What's in effect — profile, toggles, where each key comes from — is shown by aluy config.
